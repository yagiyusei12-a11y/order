# order — 追加機能の優先順位と境界（決定メモ）

この文書は「追加機能の提案」に基づき、**優先軸**と**実装時の境界（スコープの線引き）**を一度に固定するためのものです。

## 優先軸の決定（2026-05-10）

**第1優先: リスク低減・店内運用の安定**

- スタッフ人数や交代が増えるほど、権限のない操作や事後説明のない変更がボトルネックになるため、**スタッフ権限（ロール）**と**監査ログ**を他の大型機能より先に着手する方針とする。
- **売上・バックオフィス効率**（レポート CSV・期間指定）は第2に並べる。
- **法令・客向け安心**（アレルゲン構造化）は、メニューマスタと表示ポリシーの合意が必要なため、権限基盤の次か、店舗と要件が固まり次第で並行してよい。

## Phase 1 — スタッフ権限（ロール）

| 項目 | 境界 |
|------|------|
| **誰が使う** | 店舗スタッフ全員。ログイン単位で `staff` / `manager`（名称は実装で調整可）を付与。 |
| **manager のみ** | 会計上の敏感操作（例: 支払取消・大きな割引・一部設定変更・スタッフアカウント作成/削除）。確切な画面・API は実装時にルート単位で列挙する。 |
| **staff でも可** | 日常オペ（注文受付、キッチン更新、卓・会計の通常フロー、閲覧系レポートは要検討）。 |
| **外部連携** | **不要**（DB と JWT／セッション内のクレームのみ）。 |

## Phase 1 — 監査ログ

| 項目 | 境界 |
|------|------|
| **記録対象（初期）** | 伝票・支払まわりの修正、権限の効く操作、ログイン失敗の一定回数など、トラブル時に説明が必要なイベント。 |
| **誰が見る** | **manager** 以上（または将来の「本部閲覧」が入るまで manager のみ）。 |
| **外部連携** | **不要**。アプリ内一覧と CSV エクスポート程度までを想定。長期保管は DB バックアップ運用に依存。 |

## その他提案機能の境界（短文）

### 予約／事前注文（キッチン「予約注文管理」）

- **誰が使う**: 厨房スタッフが主。受付で登録された予約・ネット注文と **同一の注文 ID／状態** で繋ぐ。
- **境界**: 開店前キューと当日オーダーの **表示分離** まで。決済 gateway は既存の取り込みに合わせる。**外部連携** は予約チャネル次第（先行はアプリ内データのみ）。

### テイクアウト「準備できました」通知

- **誰が使う**: お客（連絡先は注文時に取得済みの場合のみ）。
- **境界**: 送信タイミングは「店舗が受け渡し可能にした時」の **明示アクション** をトリガにする（誤送信防止）。
- **外部連携**: **必須**（SMS / メール / LINE のいずれか）。店舗ごとにテンプレと送信可否を設定。

### レポート CSV・期間指定

- **誰が使う**: 店長・事務、将来は本部。
- **境界**: 集計単位（日／月）と税・割引の取り扱いを **画面またはヘッダ行で明示**。個人情報列は **マスクまたは出力しない** オプション。
- **外部連携**: **不要**（ダウンロードファイル）。会計ソフト取り込みは CSV 形式の固定仕様で対応。

### メニュー・アレルゲン構造化

- **境界**: マスタは **タグ列挙**（例: 特定原材料8品目）。表示は **フィルタ + 注文前の注意文** まで。医療的判断は行わない文言にする。
- **外部連携**: **不要**。

### 在庫ダッシュボード

- **境界**: 既存の `stockQty` / しきい値の **読み取り専用一覧** と、ピーク時向け **一覧から販売停止** まで（上書きルールは権限と監査に紐づける）。
- **外部連携**: **不要**。

### ゲスト側・待ち時間の見える化

- **境界**: **目安**表示（キッチンタイマー等からの推定）。確約表現は避ける。
- **外部連携**: **不要**。
